# サーベイ③ コンペ戦略（勝ち筋とロードマップ）

> [`01_input-features.md`](./01_input-features.md)（特徴量）と [`02_estimation-methods.md`](./02_estimation-methods.md)（手法）の調査を踏まえ、本コンペで上位を狙うための**具体戦略**を示す。
> ゴール：静止衛星16バンド（直近30分・最大3フレーム）→ GPM-IMERG 降水量、**RMSE 最小化**、地上レーダー非依存・地域汎化・遅延ゼロ。

---

## 1. このコンペの本質（3行）

1. **ゼロ過剰 × ロングテール**の回帰。RMSE は「無降水で正しく 0」と「強雨を大外ししない」の両立で決まる。
2. **衛星バンドのみ**勝負。古典運用アルゴリズムが頼る NWP（PW/RH）・DEM は**外部データ禁止で使えない** → WV バンドと空間/時間文脈で moisture・地形を内製するのが差別化点。
3. **3衛星・多地域**への汎化。Public は評価データの 35%、**Private（100%）が最終順位** → 過学習を避ける CV 設計が生命線。

---

## 2. 制約の再確認とその含意

| 制約 | 含意・対応 |
|---|---|
| 外部データセット禁止 | NWP・DEM・追加衛星不可。**WV マルチバンド＋空間/時間特徴**で代替。位置・時刻・天頂角・衛星種別は提供メタから計算可で利用可 |
| RMSE 単一指標 | 大外し強雨と無降水バイアスの両方を潰す。閾値・負値クリップ・較正で詰める |
| Public 35% / Private 100% | LB 過信は禁物。手元 CV を信頼し、Public との乖離を監視 |
| 提出上限 = 人数×5回/日 | 実験は手元 CV 中心。提出は仮説検証に限定 |
| 入賞コードは3モジュール構成 | `preprocess_*.py` / `train.py` / `predict.py` に分離。最初からこの形で書く |
| ライセンス（CC0/MIT/Apache 等） | 事前学習重みを使うなら**ライセンス確認必須**。timm の多くは OK だが要個別確認 |

## 3. 推奨ロードマップ（段階的に積む）

各フェーズで「手元 CV と LB の関係」を確かめながら積み上げる。早期に**エンドツーエンドの提出パイプライン**を通すことを最優先（スコアより先に「提出できる」状態を作る）。

### Phase 0 — EDA & パイプライン疎通（最優先）
- 降水分布を把握：**ゼロ率**、有雨時の分布（log でどれだけ対称化するか）、強雨の裾。
- 層別統計：**衛星別**（Himawari/GOES/Meteosat）、**地域別**、**昼夜別**、**季節別**で分布が違うか。
- 入力とターゲットの**格子・解像度の対応**（リサンプリング方法）を確定し固定。
- **ダミー提出**（全画素 0、または全画素平均）で RMSE 基準値を取得し、提出フォーマットを確定。

> 重要な基準値：「全画素 0」と「全画素=訓練平均」の RMSE は、以後の改善を測る原点。ゼロ過剰ゆえ「全 0」は意外に強いことが多い。

### Phase 1 — 物理ベースライン
- IR 窓 Tb（10.4/11.2μm）単独の **Tb→RR べき乗則**（Auto-Estimator 型）を当てはめ、CV で係数フィット。
- これで「単バンドでどこまで行くか」を把握。以降の多バンド・空間特徴の上乗せを定量化する基準にする。

### Phase 2 — GBDT 2部モデル（本命の堅実ベース）★
プロジェクトの `lightgbm`/`catboost` を活かす。**画素単位**回帰。
- 特徴量（§01 Tier 1）：全16バンド（波長帯正規化）＋主要 BTD＋近傍空間統計（複数窓 mean/std/min/Z）＋天頂角/太陽天頂角/昼夜/時刻/衛星種別。
- 構成：**①分類器**（P(rain>閾値)）×**②有雨回帰**（log1p 強度）。または **Tweedie 損失**で1段ゼロ過剰回帰も比較。
- 後処理：負値→0、微小値のゼロ丸め閾値を CV で最適化、強雨の過小評価を isotonic 較正で補正。
- 産物：**feature importance** で効く特徴を特定 → Phase 3 の CNN 設計と特徴選択に還元。

> なぜ GBDT を先に：実装/学習が速く、3衛星混在・欠測に強く、解釈性が高い。**短期間で強い LB と知見**が得られ、以後の DL 投資の方向を決められる。

### Phase 3 — CNN / U-Net（空間文脈を自動学習）★主力
- 画像対画像回帰。ROI を固定サイズパッチに切り、マルチチャネル（16バンド）入力。
- **2段階**（Oya 流：分類 U-Net＋log 回帰 U-Net）で不均衡に対処。または単一 U-Net＋重み付き損失。
- 損失：重み付き MSE / 逆密度重み（LDS）/ focal（分類側）。log ターゲット。
- 欠測・スキャン端・昼夜の**マスク**を入力チャネル＋損失マスクで明示。
- 参考数値：GOES-16→IMERG-ER の U-Net で RMSE 0.46 mm/h・CSI 0.53（低強度良好/高強度は不均衡で苦戦）→ **高強度対策（重み・較正）が差を生む**。

### Phase 4 — 時間情報の取り込み（3フレーム）
- まず軽量に：**3フレームをチャネル方向に連結**して 2D-CNN／GBDT 特徴に dTb/dt（冷却率）・3フレーム min/mean を追加。
- 余力があれば **ConvLSTM / 時間 U-Net** で本格的に時間発展をモデル化（終盤の伸びしろ）。
- 冷却率は対流発達の強い signal（30–60分リード）。移流補正やセル追跡と併用すると頑健。

### Phase 5 — アンサンブル & 仕上げ
- **GBDT × CNN のアンサンブル**（加重平均／スタッキング）。系統の異なるモデルの混合は RMSE に効く。
- seed 平均、衛星別モデルと統一モデルのブレンド。
- 後処理の最終調整（ゼロ閾値、較正、強雨クリップ上限）。

---

## 4. モデル戦略の要点

### 4.1 2段階（hurdle）は事実上必須
ゼロ過剰のため、「降る/降らない」を分けてから強度を回帰する構成（Oya, HRA, SCaMPR, Hurdle-IMDL）が一貫して強い。単一回帰でゼロも連続も当てるのは不利。**分類の閾値**と**回帰の較正**を別々にチューニングできる利点も大きい。

### 4.2 衛星別 vs 統一モデル
- **統一モデル＋衛星 embedding**：データ量を最大化、地域/衛星汎化を学習で吸収。波長帯正規化が前提。
- **衛星別モデル（3つ）**：系統差を個別最適化（Oya は衛星別学習）。
- → **両方試して CV で比較**。データ量・波長差の大きさ次第。折衷（共有 backbone＋衛星別 head）も有力。

### 4.3 ターゲット変換と損失（RMSE 直撃ポイント）
- log1p 変換で回帰安定化、ただし **RMSE は元スケール**なので逆変換後の系統バイアスを isotonic/線形較正で補正。
- 逆密度重み（LDS）or 重み付き MSE で強雨の取りこぼしを抑制。
- Tweedie はゼロ過剰連続値に直接適合（GBDT で簡便）。

---

## 5. 検証（CV）設計 ★Private で勝つための心臓部

Public は 35% のみ。**手元 CV を真の羅針盤**にする。リークを避ける分割が決定的。

- **GroupKFold を多軸で検討**：
  - *地域（name_location）で分割* → **地域汎化**を測れる（本コンペの主目的に直結）。最重要。
  - *時間（datetime）で分割* → 時間的リーク防止（近接時刻の画像が train/val に跨らない）。
  - *衛星で層化* → 3衛星の評価バランス。
- **近接時空間リークに注意**：同一イベントの 10 分違いフレームが train と val に分かれると過大評価。イベント/日単位でグループ化。
- **層化**：ゼロ率・強雨割合が fold 間で揃うよう層化（降水量ビンで stratify）。
- **CV と LB の対応を記録**：相関すれば CV を信頼、乖離すれば分割を疑う。
- 指標は RMSE 主軸＋補助（CSI・bias・強雨条件付き RMSE）で多面的に監視。

## 6. 後処理チェックリスト（地味だが効く）
- [ ] 予測の**負値を 0 にクリップ**（降水は非負）。
- [ ] **ゼロ丸め閾値**：小さな正の予測を 0 にする閾値を CV で最適化（無降水の系統誤差を削減）。
- [ ] **強雨の較正**：log 回帰の過小評価を isotonic/分位マッピングで補正。
- [ ] **上限クリップ**：非現実的な極端予測に物理上限を設定（過大外しの二乗ペナルティ回避）。
- [ ] **欠測フレーム**時のフォールバック（1–2 フレームでも動く設計）。

## 7. 実装・計算リソース

- **3モジュール構成**を最初から：`src/preprocess_train.py` / `preprocess_test.py` / `train.py` / `predict.py`。`conf/` に設定（Hydra 等）。
- **データ 18GB 対策**：前処理で「画素特徴テーブル（GBDT 用）」や「パッチ npz/zarr（CNN 用）」に一度だけ変換しキャッシュ。raster I/O は `rasterio`、大規模配列は `zarr`/`webdataset`。
- **サンプリング**：無降水画素を全部使うと冗長 → 学習時に**無降水をダウンサンプル**（ただし評価分布は保つ）。
- **GPU**：CNN は CUDA 11.8+。混合精度（amp）でメモリ節約。
- **再現性**：seed 固定、データ版管理、提出 zip 生成を `predict.py` に内包。
- **ライセンス**：事前学習重みは CC0/MIT/BSD/Apache のみ。使用前に確認しドキュメント化。

## 8. リスクと対策

| リスク | 兆候 | 対策 |
|---|---|---|
| Public LB 過学習 | CV↑だが Public↓、提出を Public に合わせがち | 手元 CV を信頼、地域 GroupKFold、提出回数を仮説検証に限定 |
| 地域/衛星非汎化 | 特定地域だけ誤差大 | 地域分割 CV、衛星 embedding、緯度経度の生値依存を抑制 |
| 強雨の系統過小評価 | conditional RMSE（強雨）が悪い | 逆密度重み・較正・hurdle、強雨オーバーサンプル |
| 巻雲の雨誤検出 | 無降水域で過大予測 | 空間 Z 特徴、BTD（split-window/phase）、2段階分類強化 |
| 昼夜のドメインシフト | 夜間だけ悪化 | 昼夜フラグ、夜間は IR/WV のみで成立する設計 |
| 入力/ターゲット格子ずれ | 全体に系統誤差 | リサンプリングを訓練/推論で厳密一致、視差補正 |
| 時間リーク | CV が楽観的 | イベント/日単位グループ化 |

## 10. 一言まとめ

> **「2段階（降る/降らない→強度）×多バンド（WV と BTD で moisture/微物理を内製）×空間文脈（巻雲除去）×頑健な地域 CV」** が本コンペの背骨。
> まず GBDT 2部モデルで堅実に土台を作り、U-Net で空間文脈を、3フレームで時間発展を積み増し、最後にアンサンブルと後処理（負値クリップ・ゼロ閾値・強雨較正）で RMSE を詰める。NWP/DEM が使えない分、**WV マルチバンドと空間/時間特徴の作り込み**が順位を分ける。
